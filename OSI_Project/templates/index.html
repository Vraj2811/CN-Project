PDU, SDU & Headers

PDU (Protocol Data Unit): The complete unit of data at a specific layer, including all headers added by that layer.

SDU (Service Data Unit): The data received from the layer above that needs to be processed and encapsulated.

Encapsulation Process

As data moves down through the layers:

  1. Each layer receives an SDU from the layer above
  2. The layer adds its own header (and sometimes trailer) information to the SDU
  3. The combined header + SDU becomes the PDU for that layer
  4. This PDU is passed down to the layer below as its SDU
Example: Web Page Request

For an HTTP request:

  • Application Layer: Adds HTTP headers
  • Transport Layer: Adds TCP header with port numbers
  • Network Layer: Adds IP header with IP addresses
  • Data Link Layer: Adds Ethernet header with MAC addresses and trailer with FCS
  • Physical Layer: Converts frames to electrical, optical, or radio signals

TCP Communication Lifecycle

TCP provides reliable, ordered, and error-checked delivery of data between applications.

Connection Establishment (Three-Way Handshake)

Before data can be transmitted, TCP establishes a reliable connection through a three-way handshake:

  1. SYN: Client sends a SYN packet with an initial sequence number
  2. SYN-ACK: Server acknowledges and sends its own sequence number
  3. ACK: Client acknowledges the server's sequence number

This process synchronizes sequence numbers and establishes connection parameters.

Data Transfer

Once the connection is established, data can be reliably exchanged:

  • Segmentation: Large data is broken into segments
  • Sequencing: Each segment has a sequence number for ordering
  • Acknowledgment: Receiver confirms receipt with ACK packets
  • Flow Control: Window size controls transmission rate
  • Retransmission: Lost packets are automatically resent

TCP ensures all data arrives in order and without errors.

Connection Termination (Four-Way Handshake)

When communication is complete, TCP gracefully closes the connection:

  1. FIN: Client sends FIN to indicate it's finished sending
  2. ACK: Server acknowledges the client's FIN
  3. FIN: Server sends its own FIN when ready to close
  4. ACK: Client acknowledges the server's FIN

This ensures both sides complete their transmissions before closing.
